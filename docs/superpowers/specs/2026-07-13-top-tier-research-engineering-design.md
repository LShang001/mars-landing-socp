# 顶级期刊与前沿工程升级设计

## 1. 目标

本项目将从单一标称场景的 SOCP 求解示例，升级为具备原创方法、统计验证、实时工程证据、闭环鲁棒性和完整复现链的研究系统。最终成果应同时满足：

1. 形成可由消融实验支持的算法或系统方法贡献，而非仅集成现有求解器。
2. 所有论文数字均能追溯到版本化场景、原始结果和生成脚本。
3. 在主机与至少一种目标嵌入式平台上报告可复现的时间、内存和收敛证据。
4. 在不确定性、故障和闭环重规划场景下报告统计性能及失败边界。
5. 提供可由独立研究者执行的 artifact 复现入口。

## 2. 不可破坏的项目资产

`MarsLanding/MarsLanding.c` 是作者原始、独立、人工完成的 CRS 稀疏矩阵构造与 CRS→CCS 转换实现。后续工作必须遵守以下约束：

- 不得用 CasADi、CVXPY 或机器生成矩阵替换该文件中的手写构造。
- 手写版是独立工程基准；自动版只承担模型审计和交叉验证职责。
- 对公共物理参数、矩阵尺寸和结果的一致性检查可以自动化，但不能改变手写实现的来源属性。
- 论文、README、实验清单和后续 agent 指令必须始终明确区分“C 手写版”和“自动生成验证版”。
- 任何针对手写版的优化都必须保持可审查的人工稀疏结构，并用自动版验证数学等价性。

## 3. 总体策略

采用五条工作线并行推进，但共享同一套证据门槛。实验可信度线首先定义公共数据契约；算法、嵌入式和高保真制导线随后并行产生候选结果；论文与复现线只接收通过门槛的结果。

```text
场景与证据契约
    ├── 理论与算法实验 ─────┐
    ├── 嵌入式工程实验 ─────┼── 证据审计 ── 论文表图与 artifact
    └── 高保真闭环实验 ─────┘
```

并行不等于无依赖：未统一场景定义、度量和结果格式之前，各工作线不得自行产生不可比较的论文数字。

## 4. 工作线设计

### 4.1 实验与可信度线

建立唯一的实验场景和结果契约，解决当前 Monte Carlo 脚本与论文描述不一致的问题。

核心产物：

- 版本化场景清单：标称状态、扰动分布、随机种子、样本数、求解器设置和模型版本。
- 结构化逐样本结果：输入状态、求解状态、燃耗、约束残差、终端误差、迭代数和耗时。
- 明确的成功判据：求解状态、原始/对偶残差、锥约束、质量边界和终端误差必须同时合格。
- 统计报告：成功率置信区间、条件成功率、燃耗分布、失败分类和最坏样本。
- 声明证据台账：论文中的每个定量结论对应数据文件、生成命令和提交版本。

当前 `mars_robustness.py` 的 100 样本固定种子基线为 59% 成功率、成功样本燃耗 `373.0 ± 49.9 kg`；论文现有的 `401.2 ± 8.3 kg` 与脚本采样定义不一致，必须在新契约下重新生成，不得手工选择其中一个数字。

### 4.2 理论与算法线

候选研究方向按风险从低到高推进：

1. 无量纲化与显式尺度设计，对比 ECOS 内部均衡化，报告条件估计、残差、迭代数和精度。
2. 参数化重复求解和热启动，评估闭环相邻制导周期中的时延与失败率。
3. 固定稀疏结构、AMD 排序和符号分解复用，区分装配、分析、数值分解和迭代耗时。
4. 自由终端时间外层优化或参数化搜索，量化可行域与燃耗收益。
5. 鲁棒约束、机会约束或故障场景约束；只有能够保持可解释凸结构或给出收敛证据时才进入主论文。

每个候选方法必须包含：未修改基线、单因素消融、组合方法、至少三种问题规模、统计重复和失败案例。仅有速度提升而无数值等价性，或仅有单场景改善，不能构成论文贡献。

### 4.3 嵌入式工程线

工程目标是可测量的确定性执行，而不是根据桌面处理器推算 MCU 性能。

核心产物：

- 分阶段计时：矩阵更新、`ECOS_setup`、`ECOS_solve`、控制提取和端到端周期分别测量。
- 稳健统计：预热、CPU/时钟设置、样本数、p50/p95/p99/最大值和异常值规则明确记录。
- 内存证据：栈、堆、静态区、工作区峰值和 LDL 填充量来自构建映射或运行测量。
- 静态工作区方案：逐步减少运行时动态分配，并验证重复求解无内存增长。
- 目标平台矩阵：主机基线、ARM Linux SBC、具备足够双精度能力的 MCU/SoC 或 RISC-V 平台。
- HIL 接口：固定周期输入导航状态，输出推力命令、状态码、求解时延和降级决策。

任何硬件结论必须标明处理器、频率、编译器、选项、功耗模式、数值精度和计时范围。未实测的平台只能写为候选平台或工程预算。

### 4.4 高保真制导线

采用逐级增加复杂度的验证阶梯：

1. 标称开环 SOCP 与独立求解器一致性。
2. 初始状态和导航估计误差 Monte Carlo。
3. 推力幅值/方向误差、质量误差、重力和大气参数扰动。
4. 单发动机降额或失效、传感器异常和求解超时。
5. 闭环重规划、可行域监视、软终端约束和安全降级。
6. 在前三层证据稳定后评估 6-DOF、姿态耦合、地形与障碍约束。

每一级使用前一级相同的结果契约，并增加相应误差和安全指标。模型复杂度升级必须说明新增状态、控制、约束、可观测量和求解成本，不能只报告成功轨迹。

### 4.5 论文与复现线

论文不再维护人工录入的实验数字。表格和数据图从审计后的结构化结果生成，示意图可继续由 TikZ 维护。

artifact 至少包含：

- 环境和依赖版本清单。
- 快速正确性验证、完整统计实验和论文全量复现三个入口。
- 原始结果、聚合结果、表图生成和论文编译的依赖链。
- 预期运行时间、硬件要求、随机性说明和失败诊断。
- 手写矩阵实现与自动模型之间的结构和数值一致性报告。

论文贡献将在证据形成后重写。暂定目标是“一项结构化重复求解方法 + 一套嵌入式确定性证据 + 一套闭环鲁棒性评估”，而不是把所有候选方向都包装为贡献。

## 5. 公共数据流

所有实验遵循以下数据流：

1. 场景清单生成确定的样本标识和输入参数。
2. 求解适配器读取同一场景，调用手写 C、自动 C 或 Python 验证路径。
3. 每次运行写出不可变逐样本记录，不在运行阶段只保留聚合值。
4. 独立聚合器计算统计量、置信区间和失败分类。
5. 声明审计器校验论文引用的数据集与提交版本。
6. 图表脚本只读取审计后的聚合结果。

运行失败也必须产生记录，包含错误类型、状态码和已有残差，禁止静默丢弃失败样本。

## 6. 验证门槛

### 正确性门槛

- 标称燃耗保持约 400.7 kg，且手写版与自动版满足预设的目标值和约束残差容差。
- 公共参数、维度、锥顺序、矩阵符号与行序差异均有机器可读检查。
- 手写文件的保护规则有自动检查，防止构建目标误指向生成数据。

### 统计门槛

- 探索性实验和论文确认性实验分离，后者固定方案、种子和样本量。
- 成功率同时报告分子、分母和置信区间。
- 性能比较使用重复测量、分位数和效应量，不只报告单次均值。
- 对不可行、数值失败、超时和物理约束失败分别计数。

### 工程门槛

- 主机与目标硬件数字不可混用。
- 实时性结论必须包含最坏观测值及测量条件。
- 静态内存和无分配声明必须由链接映射、分配计数或重复运行证据支持。
- HIL 结果必须包含超时与不可行时的安全降级行为。

### 论文门槛

- 每个定量结论都有证据台账条目。
- 所有图表可由仓库命令重建。
- 论文不包含未验证的处理器性能、条件数、加速比例或因果解释。
- 局限章节明确区分已完成、正在研究和未来工作。

## 7. 阶段划分与并行关系

> **状态（2026-07-13）**：阶段 A 已完成，完成审计见
> [`docs/evidence/stage-a-audit.md`](../../evidence/stage-a-audit.md)。阶段 B、C、D
> 仍为进行中/未来工作；阶段 A 完成不表示本项目长期目标已经完成。

### 阶段 A：可信实验基线

实验线建立场景与结果契约；论文线建立声明台账；其他三条线只进行基线测量。完成条件是当前论文主要数字均可追溯，Monte Carlo 矛盾已消除。

### 阶段 B：候选方法与平台基准

算法线并行评估尺度、热启动和结构复用；工程线建立主机与首个目标平台基准；高保真线加入执行与导航扰动。完成条件是至少一个候选方法通过消融，并获得首个非桌面平台实测数据。

### 阶段 C：闭环与安全验证

实现闭环重规划、可行域监视和降级策略，执行大规模确认性 Monte Carlo 和故障注入。完成条件是成功/失败边界、实时性和安全响应均有统计证据。

### 阶段 D：投稿与 artifact 冻结

冻结模型、实验协议和数据集，重写贡献与结论，完成一键复现和独立复核。任何阶段 D 之后的算法修改都触发受影响实验重新运行。

## 8. 第一阶段优先级

第一阶段不先改求解器，而是修复研究基础设施：

1. 定义场景和逐样本结果 schema。
2. 重构 Monte Carlo 使采样定义、结果和论文完全一致。
3. 建立约束残差与终端误差审计。
4. 建立 benchmark protocol 和原始计时结果格式。
5. 建立论文声明证据台账。
6. 为手写矩阵资产增加来源与不替代规则的自动保护。

这些工作完成后，五条工作线即可在可比较、可追溯的基础上真正并行。
